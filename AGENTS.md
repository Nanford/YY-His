# 老年健康智能评估与干预系统 Demo

## 项目概览

**项目名称**：老年健康智能评估与干预系统 Demo（YY-Demo）

**核心目标**：系统代替医护人员完成老年患者健康信息采集（数字医生语音对话），自动完成标准化评估并生成评估标签，再基于知识图谱映射推荐个体化干预方案。Demo 阶段的唯一目标是跑通完整流程：**基础信息录入 → 智能化信息采集 → 采集信息分析和评估 → 干预方案推荐**。

**使用者**：医生（录入、干预方案审核确认）+ 老年患者（大屏语音交互，**可自助建档并直接开始评估，不必等医生先录入**；问答完成后可在同一大屏直接查看评估报告与候选干预方案，无需等待医生前置确认）。两端目前均为独立入口（`/doctor`、`/patient`），Demo 阶段不做真实登录鉴权（2026-07-14 已与用户确认，正式版再加）。患者自助建档只收姓名/性别/年龄（必填）+ 测量数据（选填），并可自选评估内容（四量表多选，默认勾 FRAIL+跌倒；2026-07-15 修订，覆盖当日早先"固定 FRAIL+跌倒"的锁定口径）——含舌象/测量题的量表（MNA-SF/中医体质）在选项上如实提示"需医生补齐后出报告"，选了则答完先落 `awaiting_doctor`、由医生补录后出报告（优雅降级，见下方"已知的坑"）；医生仍可在 `/doctor/patients/new` 录入完整档案（含身份证/手机/住址/住院号/门诊号）并勾选任意量表——两条入口共用同一套校验规则（`src/lib/assessment/patient-intake.ts`），产生的患者/会话完全同构，医生审核干预方案这一关不因入口不同而跳过。

**当前阶段**：Demo / 探索。演示优先，但评分引擎、数据模型按正式产品标准做（后续会持续增加量表、标签和映射）。

## 技术栈与外部服务（已确认，勿再更换）

| 项 | 结论 |
|---|---|
| 框架 | Next.js (App Router) + TypeScript + Prisma + SQLite + Tailwind，单一代码库 |
| 回答归一化 LLM | DeepSeek API（判断回答是否满足题意 + 解析为标准答案，JSON mode） |
| TTS | 豆包 TTS（火山引擎语音合成），服务端调用 + 文本哈希缓存 |
| ASR | 火山引擎 ASR（与 TTS 同平台同密钥体系） |
| 数字医生形象 | 商用数字人 SDK（首选火山引擎虚拟数字人）；`AVATAR_MODE=sdk\|fallback` 可切换降级 2D 形象 |
| 输入模式 | 语音转文字确认 / 语音直答 / 文字输入 / 大按钮快捷作答，四种并存 |

**医学口径（用户已拍板）**：体质"倾向是"（9～10 分）与平和质"基本是"**触发干预推荐**，视同正式体质标签查映射；评估报告中仍标注"（倾向）"字样以示区分。

## 硬约束（不可违背）

1. **PII 本地化**：姓名、身份证号、手机号、住院号、门诊号、住址等直接身份信息只能存本地 SQLite，禁止出现在任何第三方云端请求中（DeepSeek、火山均只允许携带患者唯一编号）。出网前必须经过 `src/lib/providers` 层的字段过滤，且有单元测试把关。
2. **评分确定性**：自然语言回答先归一化为标准答案/分值，再由**纯函数评分引擎**严格按《量表题目_Demo.txt》计算。大模型只做语言理解（预生成文案的选择填充、归一化回答），**绝不参与评分和判定**。
3. **禁止编造**：回答模糊 → 追问 1 次 → 仍不清标"待确认"、轮末用预生成的换说法版本复问 → 仍不清标"待人工确认"由医生补录，不得强行生成答案。
4. **全链路可追溯**：标签 → 量表得分明细 → 每题标准答案 → 原始转写/语音文件，医生可逐级下钻，修改必须留痕。
5. **干预方案必须展示全文**：不能只显示"八段锦""滋阴食养"等标签名，必须同时展示《干预标签_Demo.xlsx》中对应的完整执行方案，按运动干预 / 膳食补充 / 中医食养三大类分组。禁忌提示（`extractCautions`）必须在医生审核界面**和**患者报告页同时醒目展示（红色警示区），不能只在一端出现。
6. **评估报告生成不经医生前置审核**（2026-07-14 与用户确认的产品口径）：评估标签/得分是纯函数确定性计算，患者问答全部答完（且勾选量表没有测量/观察题缺口）即自动触发评分并生成报告，患者可直接在大屏看到，医生**不需要**先确认评估结果这一步才能让患者看到。但**最终干预方案仍必须经医生审核（删/留/调整）**——患者看到的是未经审核的候选草案（`InterventionPlan.status = draft`），页面必须清楚标注"初步方案·医生确认中"，医生确认后（`status = confirmed`）患者端才会切换显示"医生已确认"。若量表存在测量题（BMI/腹围/小腿围）或观察题（舌象等），或有"待人工确认"答案未补录，问答完成后评分会失败并回退 `in_progress`，患者端显示"需要医生协助"，走医生端 `CollectForm` 补录 + 手动点"完成采集"这条老路径（M2 起就有，未改变）。

## 核心数据流

```
患者原始语音与健康信息
  → 《量表题目_Demo.txt》评估规则（归一化 + 确定性评分）
  → 评估标签集合（多标签并存，含"倾向"级）—— 问答一答完即自动生成，患者可直接看到，无需医生前置确认
  → 《评估标签-干预标签知识图谱映射表_Demo.xlsx》（逐标签查询 → 汇总 → 去重）
  → 候选干预标签集合（连同评估报告一起展示给患者，标注"初步方案"）
  → 《干预标签_Demo.xlsx》（附执行方案全文，三大类分组展示）
  → 医生审核（删/留/调整）→ 最终干预方案（患者端标注同步切换为"医生已确认"）
```

评分触发的唯一实现在 `src/lib/assessment/finalize.ts`（`acquireFinalizingLock` + `scoreAndSnapshot`），医生端表单提交（`src/lib/actions/doctor.ts` 的 `finalizeSession`）与患者端问询完成自动触发（`src/lib/dialogue/service.ts` 的 `tryAutoFinalize`）共用同一份编排，避免评分触发逻辑在两处漂移。

| 文件（docs/source/，只读） | 作用 |

| `评估标签-干预标签知识图谱映射表_Demo.xlsx` | 17 个评估标签 → 15 个干预标签，共 75 条映射边 |
| `干预标签_Demo.xlsx` | 15 个干预标签对应的完整执行方案文本 |
| `老年健康智能评估与干预系统 Demo 需求文档.docx` | 总体需求（架构依据） |

**规则**：源文件**只读，禁止修改和覆盖**。程序运行使用 `data/*.json`：两个 xlsx 由 `npm run convert-rules` 生成，`data/scales.json`（题库+口语版/复问版文案+评分规则）从 txt 手工整理并配对照测试。医学规则变更的唯一路径：改源文件 → 重跑转换/整理 → 跑测试。禁止绕过源文件直接改 `data/*.json` 的医学内容。

### 评估标签全集（17 个）

- FRAIL：无衰弱 / 衰弱前期 / 衰弱
- MNA-SF：营养正常 / 存在营养不良风险 / 营养不良
- 跌倒：跌倒风险筛查阴性 / 跌倒风险筛查阳性
- 中医体质（9 种，可多标签并存）：平和质 / 气虚质 / 阳虚质 / 阴虚质 / 痰湿质 / 湿热质 / 血瘀质 / 气郁质 / 特禀质

### 干预标签全集（15 个，三大类）

- 运动干预：八段锦、太极拳、抗阻训练、平衡训练
- 膳食补充：均衡膳食维持、优质蛋白强化、能量与营养强化
- 中医食养：健脾益气食养、温阳食养、滋阴食养、健脾祛湿食养、清利湿热食养、活血食养、疏肝解郁食养、特禀体质个体化食养

## 评分实现要点（写代码前必读）

- **FRAIL**：5 题是/否各 1 分。0 分=无衰弱；1～2 分=衰弱前期；≥3 分=衰弱。
- **MNA-SF**：A(0-2) B(0-3) C(0-2) D(0/2) E(0-2) F(0-3)，总分 0～14。12～14 正常；8～11 存在营养不良风险；0～7 营养不良。**F 题分支**：有 BMI 用 F；无 BMI 用 F替代（小腿围 CC<31cm=0 分，≥31cm=3 分），二者取其一不叠加；两者皆缺 → 该题"待人工确认"。
- **跌倒风险**：3 题任意一题"是"→ 阳性；全"否"→ 阴性。
- **中医体质**：33 题 1～5 级计分。**特殊计分题**：第 9 题按 BMI、第 28 题按腹围、第 14 题按年感冒次数、第 17 题按过敏频率取分值。8 种偏颇体质各对应 4 题，小计 ≥11 = 是；9～10 = 倾向是；≤8 = 否。**平和质**：题 1、2、4、5、13，其中 2/4/5/13 反向计分（6 − 原始分）；总分 ≥17 且其他 8 体质均 <8 → 是；总分 ≥17 且其他 8 体质均 <10 → 基本是；否则否。
- 多个偏颇体质同时命中时**全部保留**，不强制单选。
- 第 32、33 题（舌象）可由调查员/医生辅助观察填写，医生端需支持代填。

## 系统架构

### 目录结构

```
YY-Demo/
├── AGENTS.md / CLAUDE.md     # 本文件（CLAUDE.md 为 @AGENTS.md 引用）
├── docs/source/              # 四份医学源文件（只读）
├── scripts/                  # convert-rules：xlsx → data/*.json 转换+全量校验
├── data/                     # scales.json / tag-mapping.json / interventions.json
├── prisma/                   # schema + SQLite 数据库文件（.gitignore）
├── storage/audio-cache/      # TTS 音频缓存与录音文件（.gitignore）
├── src/lib/scoring/          # 评分引擎：纯函数，无 IO，100% 测试覆盖
├── src/lib/recommend/        # 推荐引擎：查映射→汇总→去重→三类分组→附全文，纯函数
├── src/lib/dialogue/         # 会话状态机、DeepSeek 归一化、规则兜底解析
├── src/lib/providers/        # deepseek / volc-asr / volc-tts / PII 过滤 / Avatar 双实现
├── src/app/doctor/           # 医生端：录入、会话创建、代填、审核追溯、方案确认
├── src/app/patient/          # 患者端大屏：数字医生、字幕、大字体大按钮
├── src/app/api/              # asr / tts / session 等 Route Handlers（密钥只在服务端）
└── tests/                    # vitest：引擎全量用例 + PII 过滤用例
```

### 分层原则

- `src/lib/scoring` 与 `src/lib/recommend` 是本项目的**医学核心**：纯函数、确定性、不依赖数据库与网络，任何改动必须先有测试。
- 题目口语化改写与追问话术采用**预生成 + 人工可审**方式（存于 data/scales.json），不在现场实时让 LLM 自由发挥，保证不改变医学含义。
- 回答归一化：DeepSeek 优先，规则匹配兜底（是/否、数字、选项关键词）；两者都失败 → "待人工确认"。
- 患者端必须保留**文字/按钮作答兜底模式**：语音链路任何一环不可用时，演示流程仍能完整跑通。

## 已知的坑

- 数字人 SDK 开通周期不可控（商务流程）→ Avatar 双实现，降级 2D 形象保演示，切换点收敛在患者端 Avatar 组件。
- 体质第 9 题（BMI）、第 28 题（腹围）、MNA-SF F 题（BMI）/F替代（小腿围）依赖测量数据：基础信息录入含身高/体重/腹围/小腿围选填字段，缺失时相关题目走"待人工确认"，医生补录后重算。
- 现场网络不可控 → TTS 按文本哈希预热缓存 + 文字/按钮作答兜底 + 归一化规则兜底。
- 部分干预方案自带禁忌提示（优质蛋白强化-肾功能异常、活血食养-抗凝药物、温阳食养-口干烦热、滋阴食养-糖尿病不加糖），医生审核界面必须醒目展示。
- 47 题全量演示过长 → 会话创建时可勾选量表，演示默认只跑 FRAIL+跌倒。**患者自助建档现在也可自选四量表（2026-07-15 修订，默认仍勾 FRAIL+跌倒）**——不再固定锁死。医学分工靠"优雅降级"而非入口限制：语音问询流程（`state-machine.ts` 的 `askableQuestions`）恒跳过 `measurement`（BMI/腹围/小腿围，由建档测量数据换算）与 `observerAssisted`（舌象、面色晦黯、神经心理等需临床观察）两类题。FRAIL/跌倒无这两类题 → 纯自助当场出报告；MNA-SF/中医体质含这两类题 → 患者答完能答的题后 `tryAutoFinalize` 因缺项回退，会话落 `awaiting_doctor`，患者端显示"需要医生帮您确认"，由医生在患者详情页 `CollectForm` 补录观察/测量题并确认方案后报告才出现。观察题仍只由医护判断，硬约束不变；建档页在含观察题的量表选项上如实提示，患者不会被"卡住"感突袭。
- 语音联调状态（2026-07-14 真实密钥验证）：**TTS 已通**（v3 单向流式 `/api/v3/tts/unidirectional`，资源 `seed-tts-2.0`，uranus 系列音色，逐行 JSON 流拼接音频）；**DeepSeek 归一化已通**（口语命中/模糊追问均正确）；**ASR 已通**（2026-07-14 补开通后复测，`volc.bigasr.auc_turbo` 拿真实录音直连与经 `/api/asr` 路由全链路复测均返回正确转写文本）；流式识别 `volc.bigasr.sauc.*` 与端到端实时语音大模型 `volc.speech.dialog` 尚未使用（`volc.speech.dialog` 已确认账号授权，WebSocket 握手返回 101，但产品化前需先定音口径，见下一条）。
- 归一化编排原则：DeepSeek 通道可用时其结论（含 unclear）即最终结论，**规则不得复核推翻**——"我没听明白"这类表达会被关键词规则误判成"否"；规则兜底只在模型通道失败（无密钥/网络/超时）时启用。
